# 林下问路模型重构方案 V0.1

## 0. 方案目的

当前“林下问路”已经具备主诉、三问、八卦状态、路径建议、风险强度、复诊入口等基础能力，但模型仍停留在轻量版 `S₈` 诊断：三问生成八种基础组织处境。

V8 完整方案与 FSKN 数学体系要求下一步从“八种基础状态”升级为“六爻现状卦 + 双重惯性 + 三辞合参 + 路径重规划”的完整问路系统。

本方案的目标不是一次性实现全部功能，而是先建立一个稳定、可计算、可验证的模型骨架，避免后续继续出现：

- 卦名、编码、图形、解释错配；
- 复诊后报告内容变化不明显；
- 用户选择变化了，但路径、风险、建议仍像固定模板；
- 前端体验先行，底层模型不稳。

## 1. 当前模型现状

当前核心代码位于：

- `lib/zhidao-model.ts`
- `app/page.tsx`

当前模型链路：

```text
主诉输入
  → 补一问画像
  → 三问：tau / phase / delta
  → resolveDiagnosis()
  → S₈ 八种基础状态
  → 惯性卦
  → 三条路径
  → 第一行动 + 14 天路书
  → 本地存档 + 复诊入口
```

当前优点：

- 交互闭环已经跑通；
- 用户能从主诉进入诊断；
- 已有风险强度、行动强度、复诊入口；
- 已建立八卦显示校验表。

当前限制：

- 三问只能生成 `S₈`，无法形成完整六爻卦；
- “不干预趋势”只有一个，尚未区分心理惯性与事理惯性；
- 路径建议仍以模板为主，缺少爻位级行动依据；
- 复诊只是重新诊断，尚未判断“哪一爻变了”；
- 当前 `phase` 只是临时变量，不等同于 FSKN 的生成元 `g`。

## 2. 目标模型总览

目标模型采用 V8 文档定义的完整闭环：

```text
主诉锚定
  → 五区位判断
  → 区位化六问
  → 六爻现状卦 M₆₄
  → 双重惯性卦：心理卦 + 事理卦
  → 三辞合参：主观辞 + 客观辞 + 应然辞
  → 目标卦选择
  → 路径寻优
  → 五级行动工单
  → 复诊重规划
  → 收敛闸门
```

模型分层如下：

| 层级 | 名称 | 状态规模 | 用途 |
| --- | --- | --- | --- |
| L0 | 五行关系层 | C₅ | 组织关系维度本体 |
| L1 | 八卦基础状态层 | S₈ | 单个组织姿态 |
| L2 | 六十四卦情境层 | M₆₄ | 父卦 + 子卦的完整处境 |
| L3 | 变卦路径层 | P₄₀₉₆ | 任一现状到任一目标的可达路径 |
| L4 | 行动控制层 | π* | 每一步行动、复诊与重规划 |

## 3. 统一状态表设计

### 3.1 八卦基础状态表 `TrigramState`

八卦不再分散在页面组件里，而应在模型层建立唯一来源。

建议新增：

```ts
type TrigramName = "乾" | "兑" | "离" | "震" | "巽" | "坎" | "艮" | "坤";

type TrigramState = {
  id: number;
  name: TrigramName;
  symbol: string;
  lines: ["yang" | "yin", "yang" | "yin", "yang" | "yin"];
  tau: 0 | 1;
  g: 1 | 2 | 3 | 4;
  delta: 0 | 1;
  managementName: string;
  managementMeaning: string;
};
```

原则：

- 卦名、符号、三爻图形、FSKN 三变变量必须来自同一张表；
- 页面只读取 `TrigramState`，不再自行解释编码；
- 所有八卦显示必须通过统一组件渲染；
- 后续六十四卦也必须引用这张表，而不是复制卦象信息。

### 3.2 六十四卦情境表 `HexagramState`

六十四卦定义为：

```text
Hexagram = LowerTrigram × UpperTrigram
```

建议类型：

```ts
type HexagramState = {
  id: number;              // 0-63
  name: string;            // 六十四卦名称
  upper: TrigramName;      // 上卦，外部参照 / 子模体
  lower: TrigramName;      // 下卦，内部状态 / 父模体
  lines: LineState[];      // 初爻到上爻
  situationTitle: string;
  coreContradiction: string;
  managementReading: string;
};
```

显示顺序必须明确：

```text
上爻
五爻
四爻
三爻
二爻
初爻
```

内部存储顺序建议固定为：

```text
[初爻, 二爻, 三爻, 四爻, 五爻, 上爻]
```

页面显示时再反向渲染，避免上下爻错位。

## 4. 主诉区位模型

V8 文档定义五个区位，对应用户困惑的入口分类。

建议新增：

```ts
type ComplaintZoneId = "direction" | "execution" | "people" | "external" | "self";

type ComplaintZone = {
  id: ComplaintZoneId;
  name: string;
  userLanguage: string;
  keywords: string[];
  relatedLines: LinePosition[];
  diagnosisFocus: string;
};
```

五区位：

| 区位 | 用户语言 | 对应爻位 | 诊断重点 |
| --- | --- | --- | --- |
| 方向困惑 | 我不知道该往哪走 | 五爻 + 上爻 | 战略与外部参照 |
| 执行困局 | 我知道去哪，但走不动 | 三爻 + 四爻 | 运营与管理铰链 |
| 人心离散 | 团队不跟我走 | 初爻 + 四爻 | 基层活力与传导方向 |
| 外部冲击 | 环境不让我走 | 上爻 + 五爻 | 外部变化与战略承压 |
| 自我消耗 | 我自己走不动了 | 二爻 + 五爻 | 资源韧性与管理者承压 |

主诉流程：

```text
用户自然输入
  → 系统提取关键词
  → 推荐一个区位
  → 回显“我听到你说……”
  → 用户确认或改选区位
  → 进入该区位六问
```

## 5. 六问诊断模型

### 5.1 六爻与六问映射

六问不是普通问卷，而是六爻编码器。

| 问题 | 爻位 | 名称 | FSKN 变量 | 管理维度 |
| --- | --- | --- | --- | --- |
| Q1 | 三爻 | 运营效率 | τ_下 | 内部执行状态 |
| Q2 | 二爻 | 财务韧性 | g_下 | 资源与承压 |
| Q3 | 初爻 | 基层活力 | δ_下 | 一线真实动能 |
| Q4 | 四爻 | 管理铰链 | τ_上 | 上下/内外接口 |
| Q5 | 五爻 | 战略决策 | g_上 | 决策与资源配置 |
| Q6 | 上爻 | 宏观环境 | δ_上 | 外部环境 |

### 5.2 每个选项必须输出两类数据

每个选项不只给一个分值，而要输出：

```ts
type LineAnswer = {
  line: 1 | 2 | 3 | 4 | 5 | 6;
  value: "yin" | "yang";
  extremity: "none" | "old_yin" | "old_yang";
  weight: number;
  evidenceText: string;
};
```

含义：

- `value` 决定现状卦；
- `extremity` 决定心理卦；
- `weight` 决定风险强度；
- `evidenceText` 用于报告解释。

### 5.3 区位化问题模板

不同区位使用不同问法，但输出必须符合统一 `LineAnswer` 结构。

例如“执行困局”区位：

```text
Q1 三爻：执行现场现在如何？
Q2 二爻：资源是否撑得住？
Q3 初爻：基层是否有真实动能？
Q4 四爻：管理接口是否顺畅？
Q5 五爻：战略决策是否稳定？
Q6 上爻：外部环境是否可预测？
```

## 6. 六爻现状卦生成

输入：

```text
六个 LineAnswer
```

输出：

```ts
type DiagnosisResult = {
  currentHexagram: HexagramState;
  lineAnswers: LineAnswer[];
  zone: ComplaintZone;
  riskScore: number;
  evidence: EvidenceItem[];
};
```

生成规则：

```text
初爻 + 二爻 + 三爻 → 下卦
四爻 + 五爻 + 上爻 → 上卦
下卦 × 上卦 → 六十四卦现状卦
```

注意：

- 计算顺序与显示顺序必须分离；
- 存储顺序永远是初爻到上爻；
- 显示顺序永远是上爻到初爻；
- 六十四卦名称必须从 `upper/lower` 查表得出。

## 7. 双重惯性卦模型

### 7.1 心理卦

心理卦由老阴、老阳触发。

规则：

| 极端标记 | 变爻 |
| --- | --- |
| old_yang | 阳 → 阴 |
| old_yin | 阴 → 阳 |
| none | 保持不变 |

心理卦回答：

> 我想/我怕怎样？

报告口吻：

- 共情；
- 解释管理者的直觉反应；
- 明确它不一定是客观趋势。

### 7.2 事理卦

事理卦由结构惯性推演。

V0.1 不必实现完整数学公式，可先做简化版：

```text
继承下卦
根据 Q4/Q5/Q6 的权重与上下卦张力选择候选上卦
输出客观惯性趋势
```

V0.2 再引入：

- 承乘势能；
- 应位势能；
- 应力系数 λ；
- 逆天角 Θ。

事理卦回答：

> 事情不干预会怎样走？

报告口吻：

- 分析；
- 预警；
- 说明结构原因。

## 8. 三辞合参模型

三辞是报告解释层，不是算卦装饰。

| 三辞 | 来源 | 回答问题 | 报告口吻 |
| --- | --- | --- | --- |
| 主观辞 | 心理卦 | 我想/我怕怎样？ | 共情、内省 |
| 客观辞 | 事理卦 | 事情会怎样走？ | 分析、预警 |
| 应然辞 | 目标卦 | 我应当怎样？ | 建议、行动 |

V0.1 可先采用模板生成：

```ts
type InterpretationText = {
  subjective: string;
  objective: string;
  normative: string;
};
```

V0.2 再接入更完整的辞库、林辞或案例库。

## 9. 路径寻优模型

当前路径为三条：

- 顺势微调；
- 重构跃迁；
- 理想靶心。

重构后保留三条路径，但其依据要从“当前八卦状态”升级到“当前六十四卦 + 目标卦 + 风险强度”。

建议路径对象：

```ts
type StrategyPath = {
  id: string;
  name: string;
  from: HexagramState;
  to: HexagramState;
  changedLines: LinePosition[];
  cost: number;
  risk: number;
  speed: number;
  resistance: number;
  rationale: string;
  firstAction: ActionTask;
};
```

路径选择原则：

| 路径 | 优先改变 | 适用场景 |
| --- | --- | --- |
| 顺势微调 | δ / 传导方向 | 风险低、局部可动 |
| 重构跃迁 | τ / 生克时序 | 接口卡死、结构牵制 |
| 理想靶心 | g / 构型底层 | 目标或资源配置需要大改 |

## 10. 五级行动工单

目标是把报告从“建议”变成“可执行”。

五级行动：

| 层级 | 名称 | 问题 | 产品 V0.1 表达 |
| --- | --- | --- | --- |
| 1 | 战略 | 去哪？ | 目标卦 |
| 2 | 计划 | 怎么分段走？ | 中继节点 |
| 3 | 方法 | 用什么方式走？ | 路径类型 |
| 4 | 策略 | 先走哪一步？ | 14 天路书 |
| 5 | 行动 | 今天谁做什么？ | 本周工单 |

V0.1 重点实现：

- 核心战场：锁定一个爻位；
- 本周行动：1-2 个动作；
- 观测指标：一个可验证信号；
- 偏差触发器：如果没变化，下一步怎么办。

## 11. 复诊重规划模型

复诊不应只是“再测一次”，而是比较两次六爻状态。

建议新增：

```ts
type RevisitResult = {
  previous: DiagnosisResult;
  current: DiagnosisResult;
  changedLines: LinePosition[];
  riskDelta: number;
  targetStillValid: boolean;
  nextPath: StrategyPath;
};
```

复诊报告必须回答：

1. 哪些爻变了？
2. 风险变强还是变弱？
3. 原目标是否仍然有效？
4. 原路径是否需要调整？
5. 下一步行动是否要升级、降级或换方向？

### 11.1 收敛闸门

沿用 V8 文档三重闸门：

| 闸门 | 判定标准 | 产品表达 |
| --- | --- | --- |
| 目标锁定 | 连续两次目标卦不变 | 方向稳定 |
| 路径归零 | 剩余变化 ≤ 2 爻 | 不需要大改 |
| 风险转弱 | 风险分下降或负面信号减少 | 进入稳态 |

## 12. 前端页面重构建议

### 12.1 诊断页

从当前：

```text
主诉 → 补一问 → 三问
```

升级为：

```text
主诉 → 区位确认 → 六问
```

### 12.2 报告页

从当前：

```text
现状判断
判断依据
如果不干预
默认推荐路径
第一行动
14 天路书
```

升级为：

```text
六爻现状卦
六问证据
心理卦 / 主观辞
事理卦 / 客观辞
目标卦 / 应然辞
三条路径
五级行动工单
复诊入口
```

### 12.3 方法体系页

保留现有八卦校验表，并新增：

- 五区位说明；
- 六爻结构说明；
- 心理卦 / 事理卦说明；
- 一步一爻重规划说明。

## 13. 开发迭代建议

### 迭代 1：模型表与状态统一

目标：

- 建立 `trigramStates`；
- 建立 `hexagramStates`；
- 把当前八卦显示从页面组件迁移到模型层；
- 加入自动校验：八卦名称、符号、lines 必须一致。

预计任务数：2-3 次。

### 迭代 2：区位化主诉

目标：

- 五区位模型；
- 主诉语义匹配；
- 用户确认或改选区位；
- 区位影响后续问题。

预计任务数：2 次。

### 迭代 3：六问诊断

目标：

- 三问升级六问；
- 每问输出 `LineAnswer`；
- 生成六爻现状卦；
- 报告页显示六爻结构。

预计任务数：3-4 次。

### 迭代 4：双重惯性与三辞

目标：

- 心理卦；
- 事理卦；
- 主观辞、客观辞、应然辞模板；
- 报告页三辞模块。

预计任务数：3 次。

### 迭代 5：路径与行动工单

目标：

- 路径改为六十四卦级别；
- 输出锁定爻位；
- 生成本周行动工单；
- 14 天路书升级为“策略 + 行动”。

预计任务数：3 次。

### 迭代 6：复诊重规划

目标：

- 比较前后六爻；
- 标记变化爻；
- 重新判断目标与路径；
- 输出下一轮行动。

预计任务数：3-4 次。

## 14. 下一步建议

建议下一步先做：

> 模型表与状态统一 V0.1

具体内容：

1. 从页面中抽出 `trigramMap`；
2. 在 `lib/zhidao-model.ts` 中建立唯一八卦状态表；
3. 所有页面显示从模型层读取；
4. 新增一个简单校验函数，确保：
   - 八卦名称唯一；
   - 符号唯一；
   - lines 唯一；
   - id 与 name 映射稳定。

这一步是“地基”。只有地基稳了，后续六问、六十四卦、复诊重规划才不会继续发生错配。
